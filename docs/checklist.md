# 🛡️ LalaSynth Survival Checklist (v1.0)

Этот чек-лист предназначен для обеспечения стабильности приложения при резком наплыве пользователей («Эффект Хабра») на инфраструктуре с ограниченными ресурсами (**512MB RAM**).

---

## 🌍 Уровень 1: Сетевая защита (Cloudflare)
*Предотвращение падения сервера от избыточных HTTP-запросов.*

- [ ] **Orange Cloud Active**: Убедиться, что проксирование включено (скрытие реального IP сервера).
- [ ] **Cache Everything**: Настроить Page Rules для кэширования всего содержимого `/assets/`, `/static/` и неизменяемых JS-бандлов.
- [ ] **Rate Limiting**: Установить лимит (например, 60-100 запросов в минуту на IP) для API-эндпоинтов поиска.
- [ ] **WAF Rules**: Включить базовую защиту от ботов и проверку целостности браузера (Browser Integrity Check).

## 🚀 Уровень 2: Backend-оптимизация (ElysiaJS & Bun)
*Экономия памяти и процессорного времени на стороне сервера.*

- [ ] **Binary Protocol**: Реализовать 4-байтовую упаковку для WebSocket (Uint16 PlayerID, Uint8 WordIndex, Uint8 Status).
- [ ] **Body Limit**: Установить жесткий `maxBodyLength: 10240` (10KB) в конфигурации Elysia.
- [ ] **LRU Memory Management**:
    - [ ] Установить `maxSize` для кэша комнат (рекомендуется не более 200MB).
    - [ ] Настроить автоматический `ttl` (Time To Live) для удаления пустых или неактивных лобби.
- [ ] **Strict Validation**: Использовать TypeBox для обрезки слишком длинных ников или поисковых запросов.
- [ ] **GC Tune**: Запускать Bun с флагом оптимизации памяти, если это поддерживает хостинг.

## 🎵 Уровень 3: Аудио и Внешние API
*Отказоустойчивость при работе с Invidious и LRCLIB.*

- [ ] **Invidious Fallback**: Создать массив из 3+ рабочих инстансов Invidious для автоматической ротации при ошибках 429/500.
- [ ] **LRC Database Cache**: Реализовать сохранение текстов песен в Turso DB. Мастер-сервер не должен запрашивать текст у LRCLIB, если он уже есть в базе.
- [ ] **Client-Side Synthesis**: Вся обработка звука (Tone.js, эффекты) должна происходить в браузере. Сервер передает только тайминги.
- [ ] **Pre-race Buffering**: Блокировать начало гонки до тех пор, пока у всех участников не прогрузится аудио-поток (`canplaythrough`).

## 📦 Уровень 4: Frontend & Assets
*Минимизация нагрузки на клиент и снижение потребления трафика.*

- [ ] **Lottie/Spine Optimization**: Проверить, что анимации монашки оптимизированы и не вызывают утечек памяти при длительной игре.
- [ ] **Code Splitting**: Разделить бандлы SvelteKit, чтобы тяжелые библиотеки (Tone.js) грузились только на странице игры.
- [ ] **CSS Animations**: Использовать `will-change: transform` для прогресс-баров, чтобы вынести отрисовку на GPU и не нагружать CPU браузера.

## 📉 Уровень 5: Мониторинг и План «Б»
*Что делать, когда ресурсы на исходе.*

- [ ] **Health Endpoint**: Создать роут `/debug/mem` для отслеживания `process.memoryUsage()` в реальном времени.
- [ ] **Kill-Switch**: Добавить флаг в конфиг для мгновенного отключения создания новых комнат (режим "доигрывания").
- [ ] **Static Mirror**: Подготовить статический JSON с топ-100 песнями на случай, если база данных Turso достигнет лимитов.

---

### Критические лимиты для инстанса 512MB
| Метрика | Порог предупреждения | Действие (Action) |
| :--- | :--- | :--- |
| **RSS Memory** | 400 MB | Принудительная очистка LRU-кэша |
| **Event Loop Lag** | > 100ms | Увеличение интервала синхронизации WS (100ms -> 300ms) |
| **Active Connections** | 1000 | Включение очереди на вход в лобби |
